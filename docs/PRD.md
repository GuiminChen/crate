# CRATE 产品需求说明（PRD）


| 属性  | 说明                          |
| --- | --------------------------- |
| 产品  | CRATE — 由 LLM 辅助的个人知识库「编译器」 |
| 读者  | 产品、工程、研究型用户                 |
| 版本  | 0.1                         |
| 状态  | 草案，供评审                      |


技术实现与设计见 [technical-design.md](technical-design.md)。

---

## 1. 背景与问题

研究者与工程师在长期主题上会积累大量分散材料（论文、网页、仓库说明、数据集、图片）。仅靠文件夹 + 笔记 + 手工链接，**上下文连贯、可导航叙事**的成本很高。另一极端是「先建向量库 + RAG」：工程与调参成本前置，且未必匹配 **人本可读** 规模（例如几十篇量级、约数十万词）。

一条务实中间路线：**把「原材料」与「编译后的维基」分离**，由 LLM **增量维护**互联的 Markdown（摘要、反向链接、概念词条），人在 **Obsidian 类工具**中观测；问答以 **落盘文件**（报告、幻灯、图）交付，并 **回灌** 知识库，再用周期性 **Lint** 提升一致性。

### 1.1 要解决的问题


| 问题         | 简述                          |
| ---------- | --------------------------- |
| **知识碎片化**  | `raw` 与可导航叙事脱节              |
| **维护成本**   | 人工维护双向链接与概念索引难以持久           |
| **问答不可沉淀** | 会话输出散落在聊天里，无法复利             |
| **规模与检索**  | 小规模可「读全文 + 索引」；超规模需可控升级到检索层 |
| **事实与可追溯** | 模型陈述需能回到 `raw` 与引用          |


### 1.2 非目标（本 PRD 不承诺）

- 替代企业级多人协作知识图谱产品。
- 在无垂直验收流程下保证医学/法律等专业场景的合规结论。
- 1:1 复现某位博主个人的脚本与目录命名。

---

## 2. 愿景与目标

### 2.1 愿景

把 token 从 **一次性对话** 迁移到 **可版本化的知识工件**：磁盘上的 Markdown 维基 + 可追溯 `raw` + 可渲染的多模态产出。

### 2.2 业务目标（可度量，建议基线）


| 指标       | 说明                                               | 建议基线          |
| -------- | ------------------------------------------------ | ------------- |
| **知识复利** | 每次重要问答产生至少 1 个可归档工件                              | MVP 可打日志统计    |
| **导航效率** | 从概念词条到 `raw` 的路径 ≤ N 跳（由链接策略保证）                  | N≤3（目标）       |
| **一致性**  | Lint 可检出孤立页、断裂链接、摘要与 raw 明显冲突等                   | 周期内可量化条目数     |
| **规模窗口** | 在 **≤~~100 篇编译文、≤~~400K words（英文当量）** 量级下端到端问答可用 | 超出需触发「检索增强」策略 |


### 2.3 用户目标

- **研究者**：快速形成主题维基，复杂问题可追溯材料。
- **工程师**：技术调研资产可沉淀，减少重复读源码/文档。

---

## 3. 用户与核心场景

### 3.1 用户画像

- **P1 深度个人用户**：单 vault、单机器，接受 CLI/Agent 维护维基。
- **P2 小团队（可选 V2）**：共享远端片段（需权限与冲突策略，本 PRD 仅预留）。

### 3.2 核心用户故事

1. **作为**研究者，**我希望**把新论文与网页剪藏进 `raw`，**以便**系统自动更新相关概念与反向链接。
2. **作为**用户，**我希望**用自然语言提问，**以便**得到「带引用路径」的答案和一份可保存的 Markdown 报告。
3. **作为**用户，**我希望**定期运行「健康检查」，**以便**修复断裂链接、补全缺失摘要、发现可写的新词条。
4. **作为**用户，**我希望**在维基变大后仍可控成本，**以便**在必要时启用全文检索/向量检索而不推翻现有结构。

## 4. 产品范围

### 4.1 MVP（建议）


| 模块           | MVP 能力                                                            |
| ------------ | ----------------------------------------------------------------- |
| **Vault 布局** | 约定 `raw/`、`wiki/`（或等价命名）、索引类文件、可选 `AGENTS.md` 等                   |
| **入轨**       | 手工/剪藏导入；图片与 md 相对路径规范；可选「监视目录」触发增量编译                              |
| **编译**       | LLM 增量：摘要、概念文、wikilinks、主题目录页                                     |
| **观测**       | 与 Obsidian 兼容的 Markdown 约定（不强制捆绑 Obsidian）                        |
| **问答 Agent** | 工具：`read_file`、搜索（朴素 grep/ripgrep 或可选 CLI）、写入限 `wiki/`；限制 Bash 范围 |
| **输出**       | 生成 md；可选 Marp/matplotlib 管线（V1 可强化）                               |
| **回流**       | 约定输出落盘路径与元数据（front-matter：`source_query`、`created_at`）            |
| **Lint**     | 规则集 v0：断链、孤立 raw 未被引用、摘要长度异常、概念重复                                 |


### 4.2 V1 增强

- **检索分层**：全文索引 + 可选向量段落检索；保留「索引 md 由模型维护」路径。
- **朴素搜索服务**：小型 HTTP + CLI，作为 Agent 工具注册。
- **自动编译触发**：`raw` mtime 变化队列 + 防抖。
- **可视化**：图表模板库、与 Marp 主题一致。

### 4.3 V2 / 展望

- **问题级编排**：为单次提问生成短命维基 → 多轮 Lint → 长报告；作业隔离与清理策略。
- **多 Agent 角色**：检索员 / 写手 / 校对 / 制图（需编排与成本上限）。

### 4.4 明确不做（本期）

- 多人实时协同编辑冲突合并（OT/CRDT）。
- 托管 SaaS 账号体系（除非另立项目）。
- 内网文档自动爬取（版权与合规）。

## 5. 功能需求


| ID    | 名称        | 描述                                            | 优先级 |
| ----- | --------- | --------------------------------------------- | --- |
| FR-01 | Vault 脚手架 | 初始化标准目录、README、约定文档                           | P0  |
| FR-02 | Raw 入轨    | 支持拖入/剪藏/复制；图片与 md 相对路径规范                      | P0  |
| FR-03 | 增量编译      | 基于变更的 raw 子集，更新 wiki 页与 backlinks             | P0  |
| FR-04 | 全库索引      | 维护 `INDEX.md` 或分层索引（主题 → 文档）                  | P0  |
| FR-05 | 问答        | Agent 读取索引 + 按需展开正文；回答须带可追溯引用（文件名/节）          | P0  |
| FR-06 | 文件化输出     | 默认写入 `wiki/outputs/` 或用户配置路径                  | P0  |
| FR-07 | 回流        | 输出 front-matter 与回链；可选自动挂到索引                  | P1  |
| FR-08 | Lint      | 断链、重复标题、raw↔wiki 对齐检查；可选 LLM 语义冲突             | P1  |
| FR-09 | 朴素搜索工具    | CLI：`search --vault ... "query"` → 路径+snippet | P1  |
| FR-10 | 规模门闸      | 超阈值（可配：总词数、文件数）提示启用增强检索                       | P1  |
| FR-11 | 成本与配额     | 单次编译/问答 token 预算；模型路由（强模型仅用于编译与终稿）            | P2  |


### 验收要点

- **编译任务**：可重复执行；同一 `raw` 版本应对应可解释的 wiki diff（或变更日志）。
- **问答**：禁止无材料支撑的「绝对断言」句式（产品策略可配置：保守/平衡）。
- **Lint 报告**：机器可读（JSON）+ 人类可读（Markdown），便于 CI。

## 6. 非功能需求

### 6.1 性能

- 单次增量编译在「日常新增 ≤5 篇 raw」前提下，**交互可接受延迟**（目标：分钟级内，依模型与体积而定）。

### 6.2 可靠性与恢复

- Wiki 与 raw 均在 Git 管理时，支持 **回滚**；编译任务写临时文件或 `.tmp` 再原子移动（实现见技术方案）。

### 6.3 安全与隐私

- **本地优先**：密钥与 vault 不上传（除非用户显式开启云模型 API）。
- **剪藏内容版权**：用户自担；产品提示「勿自动重新发布受限内容」。

### 6.4 可观测性

- 事件：`ingest_`*、`compile_*`、`qa_*`、`lint_*`、`output_*`；可对接 OpenTelemetry（可选）。

### 6.5 兼容性

- Markdown 方言：**Obsidian wikilinks** 与 **标准 Markdown 链接** 可配置互转。

| -------------- | ----------------------------- |
| **外部 LLM API** | 需 API key；建议支持多供应商（主备）        |

## 8. 风险与假设


| 风险       | 缓解                       |
| -------- | ------------------------ |
| 模型幻觉写入维基 | Lint + 强制引用 raw；高风险页人工门禁 |
| 规模上升成本陡增 | 门闸与模型分层；引入分段检索           |
| 长周期链接腐化  | 周期 Lint；CI 钩到 git hook   |

## 10. 附录：术语


| 术语       | 含义                                      |
| -------- | --------------------------------------- |
| **raw**  | 原始材料目录，尽量保持来源可追溯                        |
| **wiki** | LLM 维护的「编译后」Markdown 图                  |
| **编译**   | 由 LLM 将 raw 转为结构化、互联的 wiki 页之过程（非传统编译器） |
| **回流**   | 问答产出再次写入 wiki，形成复利                      |
